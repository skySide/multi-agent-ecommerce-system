# 多Agent电商智能推荐系统 — 项目介绍（面试准备）

> 技术栈: Spring Boot 3.2 / Spring AI 1.0 / MyBatis-Plus / MySQL 8.0 / SimpleVectorStore / React 18

---

## 项目背景

独立开发的电商推荐与对话系统，将 LLM 以多 Agent 形式落地电商场景。核心目标是**提升用户在对话式购物中的满意度**——让推荐更准、让回答更可靠、让系统更稳定。

---

## 整体架构

Supervisor 编排器串联 3 个 Agent 串行执行，Agent 之间不直接通信，数据通过编排器中转：

```
用户请求 → 编排器
    ├── 用户画像 Agent（LLM + 行为工具 → RFM 分群/偏好提取）
    ├── 商品推荐 Agent（LLM + 商品检索工具 → 规则引擎兜底）
    └── 库存决策 Agent（LLM + 库存查询工具 → 规则兜底）
        ↓
    编排器聚合结果（含各环节耗时 + 状态）→ 返回
```

独立模块：推荐引擎（多路召回+RRF+MMR，被 Agent 内部调用）、多轮对话引擎。

---

## 核心亮点

### 亮点1: 意图识别 + 三层记忆，让对话理解更准确

**问题**：用户说"推荐一个"时系统不知道推荐什么；说"那换货呢"时"那"指代上文的退货政策。如果每次把完整对话历史传给 LLM，Token 消耗大且上下文窗口有限。

**方案**：

**意图识别**：用户消息到达后，LLM 根据最近对话历史判断意图类型，同时提取实体（类目、品牌、价格区间、商品名）。共 5 种意图：
- `recommend` → 想被推荐商品
- `product_query` → 询问某款商品详情
- `knowledge_query` → 问售后/物流/政策
- `compare` → 对比多款商品
- `chitchat` → 闲聊兜底

**三层记忆系统**：

| 记忆层 | 存什么 | 容量 | 怎么用 |
|--------|--------|------|--------|
| 短期记忆 | 对话历史 JSON + LLM 生成的摘要 | 10 轮滑动窗口 | 注入所有 LLM 调用的 prompt |
| 会话记忆 | 跨轮次累积的实体（预算、类目、品牌） | 无限制 | 新值覆盖旧值，旧字段保留 |
| 长期记忆 | `user_profile` 表中的历史偏好 | 无限制 | 对话开始时读取，失败时跳过 |

**关键机制**：
- 短期记忆 = `summary（摘要） + 最近 6 条历史（≈3 轮）`，摘要由定时任务在对话超 5 轮后生成
- 会话记忆采用 `putAll` 合并策略：上一轮说了"预算 5000"，下一轮说"推荐一个手机"，合并后仍知道预算 5000
- 长期记忆降级：读取 `user_profile` 失败时 catch 返回空字符串，不影响主流程

**效果**：用户说一次偏好就记住，不需要重复表达。意图识别结合上下文，指代消解准确率提升。

---

### 亮点2: Supervisor 编排 + 多 Agent 协同，让推荐结果更可靠

**问题**：推荐涉及画像分析、商品召回、库存校验三个环节，串行执行延迟高，且任一环节失败会导致整个链路不可用。

**方案**：

**Supervisor 编排**：编排器作为中央控制器，接收请求后依次调用各 Agent。Agent 之间不直接通信，数据通过编排器中转：

```
编排器: getUserProfile() → recommendProducts() → checkInventory()
         ↓                    ↓                    ↓
       AgentResult          AgentResult          AgentResult
```

每个 Agent 继承统一基类，返回标准结果结构（含成功/失败标记、耗时、置信度、数据载体、错误信息），编排器根据成功标记决定是否使用兜底。

**Agent 的 Tool 机制**：每个 Agent 通过 Spring AI 的 Tool 模式调用外部工具获取数据：
- 用户画像 Agent → 行为收集工具（查询用户行为、过滤误操作、返回结构化数据）
- 商品推荐 Agent → 商品检索工具（调用推荐引擎获取完整商品列表 → LLM 从中选取→输出 JSON）
- 库存决策 Agent → 库存查询工具（逐个查询库存 → LLM 汇总分析 → 输出可用列表和限购策略）

**降级兜底**（每层都有替代路径）：
- Agent 层：基类封装重试（500ms × 最多 2 次），全部失败返回降级标记，不抛异常
- Agent 内层：LLM 失败 → 规则推荐引擎（多路召回+RRF 融合+MMR 多样性）→ 热门商品兜底
- 编排器层：画像失败 → 默认画像；库存过滤后为空 → 原始列表保底
- 对话层：意图识别异常 → 闲聊兜底

**效果**：LLM 不可用时系统不挂，异常不传播到上层。每个环节失败都有替代方案，无需人工介入。

---

### 亮点3: Tool 调用权限隔离，防止 LLM 越权

**问题**：Agent 通过 Tool 机制让 LLM 调用外部工具。如果不加约束，LLM 可能跨域调用——比如营销 Agent 去查库存数据，导致数据混乱甚至越权。

**方案**：

**三层隔离**：
1. **注册隔离**：每个 Agent 只注入自己域内的工具，LLM 根本看不到其他域的工具存在
2. **指令约束**：每个 Agent 的 System Prompt 明确指定可用工具和调用顺序
3. **参数校验**：工具入口做参数校验，无效入参返回空而不抛异常，LLM 能继续推理

**效果**：从根源杜绝跨域调用。LLM 行为限制在可控范围内。

---

### 亮点4: RAG + Query 改写，让知识回答不编造

**问题**：LLM 直接回答售后政策类问题容易产生幻觉，编造不存在的政策。

**方案**：

**文档向量化**：退换货政策/配送说明/会员权益/手机选购指南 → 500 字符分块（50 字符重叠防止信息割裂）→ Embedding → 存入向量库

**检索流程**：
```
用户问"那换货呢"
  → Query 改写：结合历史中的"退货政策"，改写为"退货政策的换货规则是什么"
  → 向量库检索 Top-3
  → 文档片段注入 Prompt
  → Prompt 硬约束："知识库没有就说不清楚，不要编造"
  → LLM 基于文档内容回答
```

**Query 改写**：`rewriteQuery()` 结合对话历史和摘要，把指代不明的 query 改写为明确的问题。改写后的 query 用于向量检索，降低检索偏移。

**效果**：回答全部基于知识库文档，未覆盖的主动告知"我不清楚"。Query 改写消除指代歧义后检索命中率提升。

---

### 亮点5: 满意度可观测闭环，用数据驱动优化

**问题**：对话系统好不好不能靠感觉。不知道哪些场景满意度低、不知道优化方向。

**方案**：

**采集**：每轮对话用户可点赞或点踩。接口记录用户 ID、会话 ID、消息索引、用户消息原文、AI 回复原文、评分值。支持重复提交（覆盖更新）。

**统计**：全局满意度 = 点赞数 / (点赞数 + 点踩数) × 100%。支持按用户分组统计，评分分布按 rating 分组计数。

**针对性提升手段**：

| 用户不满的场景 | 原因分析 | 解决手段 |
|--------------|---------|---------|
| 同一问题重复问 | 系统没记住用户说过什么 | 会话记忆跨轮次合并，上一轮的预算/类目自动继承 |
| 推荐结果都一样 | 同质化严重 | MMR 多样性控制（λ=0.5），跨类目推荐 |
| 答非所问 | 没理解上下文指代 | 意图识别注入最近对话历史 + 摘要 |
| AI 编造政策 | 没有知识来源 | RAG 检索知识库 + "不知道就说不清楚" |

**效果**：满意度可追踪、可分组统计、可定位问题场景。每次反馈保留完整的消息原文，支撑离线分析。

```
多Agent电商智能推荐系统 | 个人项目 | 2026.01-2026.04

技术栈：Spring Boot 3.2 · Spring AI 1.0 · MyBatis-Plus · MySQL 8.0 · React 18

基于Supervisor编排模式构建的多Agent电商推荐与对话系统。
3个Agent（用户画像/商品推荐/库存决策）通过编排器串行协同，
集成多轮对话引擎，以提升用户满意度为目标进行全链路设计。

核心机制：

1.【意图识别+三层记忆】LLM根据最近对话识别5种意图并提取实体。
  短期记忆=摘要+最近3轮历史（定时任务压缩超5轮会话）；
  会话记忆跨轮次合并（putAll策略，旧字段保留）；
  长期记忆读取user_profile历史偏好（失败时catch不阻塞）。
  用户说一次偏好就记住，不需要重复表达。

2.【Supervisor编排+多Agent协同】编排器串行调用各Agent，数据通过
  统一结果结构中转。Agent基类封装重试+降级，失败不抛异常。
  推荐链路：LLM+商品检索工具 → 规则推荐引擎兜底 → 热门商品保底。
  4层降级（Agent层/Agent内/编排器层/对话层），LLM不可用时系统不挂。

3.【Tool权限隔离防越权】每个Agent只注入自有工具，LLM看不到其他域
  的工具。配合指令约束+参数校验，从根源杜绝跨域调用。

4.【RAG+Query改写降幻觉】知识库文档分块→向量化→检索Top-3，
  Query改写消除指代歧义后检索，Prompt硬约束"不知道就说不清楚"。
  回答基于文档不编造。

5.【满意度可观测闭环】每轮点赞/点踩 → 全局+用户满意度统计。
  跨轮次记忆合并减少重复询问，MMR多样性控制避免推荐单一，
  意图识别注入历史解决答非所问。数据驱动持续优化。
```
